# Hedge ratio и направление входа для relative-value связок

## Что исправлено

В текущей реализации коэффициент `1/2` и направление входа были слишком плотно привязаны к визуальному сравнению пройденного расстояния и к правилу `SELL leader / BUY follower`.

Для связок вроде `EURUSD` / `USDCHF` это даёт систематическую ошибку:

- коэффициент по `high-low` не равен коэффициенту хеджа;
- отрицательная корреляция не означает, что ноги надо открывать в противоположные стороны;
- для `EURUSD` / `USDCHF` рыночно-нейтральный вход чаще получается **в одну сторону по обеим ногам**.

## Новый подход

Реализация разделена на два независимых слоя.

### 1. Execution hedge ratio

Это коэффициент объёма для второй ноги.

Сначала обе пары переводятся в серию `close-to-close` PnL на 1 лот.

Для USD-котируемых и USD-базовых пар используется USD-PnL. Если символ не содержит USD ни в base, ни в quote, используется прокси через `pct_change * contract_size`, чтобы не ломать UI на кроссах.

Дальше считается:

```text
q_exec = -Cov(dPnL_1, dPnL_2) / Var(dPnL_2)
```

Где:

- `abs(q_exec)` — объём второй ноги в лотах на 1 лот первой ноги;
- `sign(q_exec)` — отношение направлений:
  - `q_exec < 0` -> ноги открываются **в разные стороны**;
  - `q_exec > 0` -> ноги открываются **в одну сторону**.

### 2. Spread model

Для сигнала на вход строится отдельная модель, которая не переиспользует `q_exec` как ценовой коэффициент.

Если отношение направлений **opposite**, то в регрессии используется обычная вторая цена:

```text
x_t = ln(P1_t)
y_t = ln(P2_t)
x_t = a + beta * y_t + u_t
spread_t = u_t
```

Если отношение направлений **same**, то вторая цена инвертируется по знаку в модели:

```text
x_t = ln(P1_t)
y_t = -ln(P2_t)
x_t = a + beta * y_t + u_t
spread_t = u_t
```

Это даёт один и тот же смысл спреда:

- `spread_z > 0` -> первая нога переоценена относительно второй -> открываем `short spread`;
- `spread_z < 0` -> первая нога недооценена -> открываем `long spread`.

## Правило направлений

### Если `side_relation == opposite`

- `long spread` -> `BUY symbol_1`, `SELL symbol_2`
- `short spread` -> `SELL symbol_1`, `BUY symbol_2`

### Если `side_relation == same`

- `long spread` -> `BUY symbol_1`, `BUY symbol_2`
- `short spread` -> `SELL symbol_1`, `SELL symbol_2`

## Что это меняет в проекте

1. Логика объёма вынесена в отдельный пакет `services/market/hedge/`.
2. Подсказка сделки теперь строится не по `leader/follower`, а по `spread_z` и `side_relation`.
3. Кнопки открытия больше не предполагают, что вторая нога всегда противоположна первой.
4. Для ручной проверки добавлен CLI-скрипт `inspect_hedge_window.py`.
5. Для быстрых проверок добавлены unit tests.

## Как проверять

### Быстрая локальная проверка логики

```bash
python -m unittest discover -s tests -p "test_*.py"
```

### Проверка на MT5 истории

```bash
python -m src.app.ui_relative_compare.tools.inspect_hedge_window \
  --symbol-1 EURUSD \
  --symbol-2 USDCHF \
  --timeframe M1 \
  --bars 1500
```

Скрипт покажет:

- rolling correlation по PnL;
- `q_exec`;
- `side_relation`;
- `spread_beta`;
- `spread_z`;
- рекомендованное направление входа.
